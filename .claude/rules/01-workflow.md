# Workflow-Regel

## Immer vor Änderungen

1. `git status`
2. relevante Dokumente lesen
3. Plan schreiben
4. Dateien nennen, die geändert würden
5. Tests nennen
6. Nicht-Ziele nennen

## Implementierung nur nach Go

Wenn der Nutzer nur fragt, bewertet oder plant:
- keine Dateien ändern
- keine Scaffold-Dateien erzeugen
- keine Dependencies installieren

## Best-Practice-Workflow

Erkunden -> Planen -> Implementieren -> Verifizieren.

Kein Direktbau bei unklarem Umfang.
